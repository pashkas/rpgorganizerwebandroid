### База знаний по задачам (UserTasks)

В проекте ведётся база знаний по всем выполненным задачам — папка `docs/UserTasks/`.
Структура:
- **`!taskIndex.md`** — главная таблица-индекс: краткое описание каждой задачи, связи между ними, ссылка на файл. **Читай первым делом** — по нему можно быстро понять, была ли похожая задача, не открывая каждый файл.
- **`NNNN.md`** (например `0012.md`) — подробное описание конкретной задачи: что было, как решили, какие файлы затронуты. Открывай только если по индексу нашёл что-то похожее.

**Зачем это нужно:** чтобы не наступать на те же грабли дважды. Если похожая проблема уже решалась — в индексе видно связь, а в файле задачи лежит готовый подход.

#### Индекс задач (`!taskIndex.md`)

Файл `docs/UserTasks/!taskIndex.md` — это главная таблица всех задач. Формат:

```
| Описание | Теги | Связанные | Файл |
|----------|------|-----------|------|
| Одно предложение — суть задачи | ФЛК, XSD, WorkflowTests | 0003, 0007 | 0009.md |
```

- **Описание** — одно короткое предложение, суть задачи своими словами. Подробности — в файле
- **Теги** — ключевые слова для быстрого поиска (подсистемы, типы файлов, номера MSG/TRN, имена классов)
- **Связанные** — номера задач, которые пересекаются по области/файлам/подсистеме
- **Файл** — имя файла с подробным описанием (только номер, `NNNN.md`)

#### Перед началом ЛЮБОЙ задачи (обязательный порядок действий):

1. **Прочитай индекс** `docs/UserTasks/!taskIndex.md` — это первый шаг, ДО любого анализа кода.
2. **Найди похожие задачи по индексу** — сравнивай по СМЫСЛУ и ОБЛАСТИ, не по точному совпадению слов:
   - Затрагивает те же файлы? Тот же процесс? Тот же тип ошибки? Ту же подсистему?
   - Если хоть по одному критерию есть пересечение — это похожая задача.
   - Если по описанию в индексе _кажется_ похожим — **открой сам файл задачи и прочитай**, описание в индексе может быть недостаточным.
   - Примеры связей: «ошибки структурного контроля MSG-файлов» ↔ «SAXParseException при валидации XML» — одна область. «Проблема с SOAP-обёрткой» ↔ «ошибка в тестовом XML» — связаны, если речь про те же файлы.
3. **Если нашёл похожую задачу — СТОП. Не лезь в код.** Скажи пользователю: «Нашёл похожую задачу XXXX_name — там описан такой-то подход. Текущая задача решается так же, или тут другой случай?» И жди ответа.
4. **Если похожих задач нет** — скажи «Похожих задач в UserTasks не нашёл» и приступай к работе.

> Лучше перестраховаться и спросить пользователя, чем пропустить связь.

#### После завершения задачи (когда пользователь скажет «ок», «готово», «отлично», «сделано» и т.п.):

**Шаг 1.** Создай файл описания в `docs/UserTasks/NNNN.md` (порядковая нумерация, 4 цифры, например `0012.md`). Вся информация о задаче — в индексе и в самом файле, имя файла — только номер.

**Что должно быть в файле:**
- **«Запрос пользователя»** — краткая цитата, с чем пользователь пришёл изначально
- Цель задачи, в чём была проблема
- Какой подход выбрали и почему именно такой
- Какие файлы и классы затронуты
- Какие побочные эффекты учтены
- **«Корректировки от пользователя»** (если были) — что уточнил пользователь и как это повлияло на решение

Чем подробнее — тем лучше. Это будущая подсказка для аналогичных задач.

**Шаг 2.** Обнови индекс `docs/UserTasks/!taskIndex.md`:
1. Добавь новую строку **в начало таблицы** (сразу после строки-заголовка и разделителя), не в конец. Свежие задачи чаще релевантны текущей работе — так они первыми попадаются на глаза при поиске похожих. Номера в колонке «Файл» при этом идут по убыванию сверху вниз — это ожидаемо.
2. Проставь связи: определи, какие существующие задачи связаны с новой
3. **Обнови связи у существующих задач** — если новая задача связана с задачей 0005, то и у 0005 в колонке «Связанные задачи» должна появиться ссылка на новую

> **Не спрашивай разрешения на правки в `docs/UserTasks/`** (ни в `!taskIndex.md`, ни в `NNNN.md`). Создание новых файлов задач, обновление индекса, правка связей, переименование, схлопывание — делай сразу, без подтверждения. Это внутренняя база знаний, а не код проекта.

#### Схлопывание задач (при 100+ записях в индексе)

Когда в индексе накопится **100+ задач** — перед добавлением новой задачи схлопни связанные цепочки:
1. Найди группы задач, связанных между собой (например 0003 → 0004, или 0008 → 0009)
2. Объедини цепочку в один сводный файл: общая суть проблемы, финальное решение, все затронутые файлы
3. Удали старые файлы, в индексе замени несколько строк на одну
4. Сообщи пользователю что схлопнул и какие задачи объединил

## Цель проекта

Геймифицированный таск-менеджер (аналог Habitica). Пользователь создаёт RPG-персонажа, настраивает характеристики, навыки, квесты и задачи. За выполнение задач прокачиваются навыки, растёт опыт и уровень персонажа, открываются награды. ToDo-лист в обёртке RPG-игры.

## Стек

- **Frontend:** Angular 9.1.13, TypeScript 3.8.3, RxJS 6.5
- **UI:** Angular Material 8 + Bootstrap 4 + Angular Flex-Layout + ngx-masonry
- **Backend (BaaS):** Firebase (Firestore + Firebase Auth)
- **Mobile:** Capacitor 5 (Android), PWA с Service Worker
- **Бэкенда на Java/Spring нет** — вся бизнес-логика на клиенте

## Команды

```bash
npm install          # установка зависимостей
ng serve             # dev-сервер (http://localhost:4200)
ng build             # сборка
ng build --prod      # продакшн-сборка
ng test              # тесты (Karma + Jasmine)
firebase deploy      # деплой на Firebase Hosting
```

## Архитектура

### Модульная структура

Проект состоит из нескольких Angular-модулей. **Важно:** при добавлении компонентов/pipe/директив нужно объявлять их в правильном модуле, а Material-модули импортировать там, где они используются.

```
AppModule (корневой, eager)
├── SharedModule (общие компоненты, реэкспорт Material-модулей)
├── PersModule (lazy: /pers) — основной функционал
├── MindMapModule (lazy: /mind-map)
└── SyncModuleModule (lazy: /sync)
```

#### AppModule (`app.module.ts`)
- Корневой, bootstrap. Содержит: MainWindowComponent, TimerCounterComponent, ProgressBarNumComponent, диалоги (LevelUpMsg, ArrSortDialog, ConfirmationDialog, TskTimeValDialog, TaskTimerComponent)
- Импортирует SharedModule, Firebase-модули, MatChipsModule, MatBadgeModule, MatIconModule, MatSlideToggleModule

#### SharedModule (`shared/shared.module.ts`)
- **Центральный модуль переиспользуемых компонентов.** Импортируется в AppModule, PersModule, MindMapModule
- Объявляет: AddItemDialogComponent, PersChangesComponent, DiaryEditParamsComponent
- Объявляет pipes: TimeValPipe, DatestringPipe, AbHardnessPipe, RequrencePipe, AbColorPipe, MainViewPipe, MasonryPipe, FilterByPropertyPipe
- **Реэкспортирует** Material-модули (Dialog, List, Input, Slider, ProgressBar, Button, Select, Checkbox), FormsModule, ReactiveFormsModule, DragDropModule, FlexLayoutModule, NgxMasonryModule
- **Новые Material-модули добавлять сюда** (imports + exports), чтобы они были доступны в PersModule и MindMapModule

#### PersModule (`pers/pers.module.ts`) — lazy-loaded
- **Основной модуль** (~30 компонентов): TaskDetailComponent, PersListComponent, AbilityDetailComponent, CharacteristicDetailsComponent, QwestDetailComponent, DiaryComponent, EnamiesComponent, LoginComponent, PersMasterComponent и др.
- Импортирует SharedModule + MatChipsModule, MatIconModule, MatBadgeModule, MatStepperModule, MatSlideToggleModule, MatCardModule, MatTabsModule, ChartsModule, OwlDateTimeModule
- Маршруты: `/pers`, `/pers/task/:id/:isEdit`, `/pers/ability/:id`, `/pers/characteristic/:id`, `/pers/qwest/:id`

#### MindMapModule (`mind-map/mind-map.module.ts`) — lazy-loaded
- MindMapComponent, MindMapOptionsComponent. Импортирует SharedModule + NgxEchartsModule, MatBottomSheetModule

#### SyncModuleModule (`sync-module/sync-module.module.ts`) — lazy-loaded
- SyncModuleComponent. **Не импортирует SharedModule** — использует только MatStepperModule, MatFormFieldModule, MatInputModule

### Управление состоянием
Никакого NgRx. Центральный «store» — `PersService` с BehaviorSubject-ами:
- `pers$` — текущий персонаж
- `currentTask$` — текущая задача
- `currentView$` — текущий вид
- `skillsGlobal$`, `qwestsGlobal$` — глобальные списки

Компоненты подписываются через `.asObservable()`.

### Доменная модель (src/Models/)
- `Pers` — персонаж (уровень, опыт, HP, золото, ранг, характеристики, задачи, квесты, награды)
- `Characteristic` — характеристика (содержит массив `Ability`)
- `Ability` — навык (содержит массив `Task`)
- `Task` — задача (значение, повторяемость, состояния, таймер/счётчик)
- `Qwest` — квест (группа задач с наградами)
- `Reward` — награда/артефакт
- `GameSettings` — абстрактный класс настроек RPG-механики, реализации подменяются через DI (`EraSettings`, `EraSettings5Lvl`)

ID генерируются через `uuid()`.

### Ключевые сервисы
- `PersService` — центральный сервис, вся RPG-логика (прокачка, опыт, задачи)
- `PerschangesService` — показ попапов изменений (до/после)
- `AuthService` — Firebase Auth (email/password, Google)
- `UserService` — работа с пользователем через Firestore
- `StatesService` — состояния задач
- `RevardService` — награды
- `EnamiesService` — враги/монстры

### Паттерны компонентов
- `ChangeDetectionStrategy.OnPush` в ключевых компонентах
- `MatDialog` для модалок
- `entryComponents` для динамических компонентов (Angular 9)
- **Иконки:** `<mat-icon>` не используется — вместо этого PNG из `assets/icons/` (create.png, del.png, up.png, down.png, right.png, edit.png и др.)

## Стиль кода TypeScript (Strict)

### Язык
- Ответы, комментарии в коде, логи и вывод в консоль — **только на русском**
- Краткие JSDoc-комментарии на русском для методов/классов (без описания параметров, только суть)

### Общие правила
- Писать максимально просто, ориентируясь на существующий стиль проекта
- Перед `return` всегда пустая строка
- Предпочитать стримы и лямбды. В лямбдах — короткие имена: `q => q.getValue()`, `doc => doc.getId()`
- Ссылки на методы НЕ используем, пишем лямбды
- Не трогать legacy-именования (`reqvirements`, `requrense`, `enamies`, `revard`) — сохранять как есть

### Именование
- camelCase для переменных и методов
- PascalCase для классов и интерфейсов
- Сохранять существующий микс стилей в полях моделей (legacy)

### RxJS
- `BehaviorSubject` как основной паттерн состояния
- Подписки через `.asObservable()` в компонентах
- `takeUntil(unsubscribe$)` для отписки в компонентах
- Операторы: `map`, `filter`, `switchMap`, `combineLatest`

## Рабочий процесс

- **Компиляцию проверяет пользователь** — не запускать `ng build`, `ng serve`, `ng test` для проверки. Пользователь сам проверит после внесения изменений.


## Legacy-особенности (не трогать)
- Опечатки в именах сущностей: `reqvirements` (не requirements), `requrense` (не recurrence), `enamies` (не enemies), `revard` (не reward в некоторых местах)
- `angularfire2` вместо `@angular/fire` в некоторых импортах
- Русские строки прямо в моделях (ранги, подбадривания, ругательства)
- `entryComponents` — необходимы для Angular 9
