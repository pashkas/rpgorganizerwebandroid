# 0003. `recountAbilMayUp` — цепочка фильтров, каждый шаг только сбрасывает `mayUp = false`

## Суть
`PersService.recountAbilMayUp(prs)` — центральная точка логики «какие навыки/перки можно прокачивать прямо сейчас». Метод разбит на нумерованные шаги (Шаг 1, Шаг 2, Шаг 3…), и инвариант жёсткий: **каждый шаг только сбрасывает `tsk.mayUp = false`, никогда не ставит в `true`**. Значит, порядок шагов задаёт приоритет ограничений, а не композицию. Шаг, который зависит от результата другого (напр. «есть ли ещё непокачанные обычные навыки»), должен считаться ПОСЛЕ того, как все предыдущие ограничения уже применены к обычным навыкам.

## Где в коде
- `src/app/pers.service.ts:3605` — `private recountAbilMayUp(prs: Pers)`, весь метод разбит на шаги с JSDoc.
- Структура (на момент 2026-04-21): Шаг 1 — нехватка ОН/ОП; Шаг 2 — потолок уровня навыка по уровню перса (`1 + floor(persLevel/10)` для обычных, `maxAbilLvl` для перков); Шаг 3 — блокировка всего остального, пока не добит «начатый» сложн-перк (`perkHardnes === 1 && 0 < value < maxAbilLvl`).
- Пересортировка характеристик/навыков (апаемое всплывает наверх) — в конце метода.

## Почему важно
Все балансные правки прокачки (задачи 0019, 0022, 0023, 0025) шли через правку именно этого метода. При добавлении нового ограничения: вставляй новый Шаг N в правильной позиции, учти, что `ch.anyMayUp` считается на Шаге 1 и позже не пересчитывается (если новый шаг блокирует последнюю апаемую ветку — сортировка покажет «есть куда качать», хотя реально нет). Не пытайся в одном шаге одновременно и поднимать, и сбрасывать `mayUp` — сломается вся логика последующих шагов.
