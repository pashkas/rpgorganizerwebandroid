# 0002. Балансные правки идут в EraSettings, не в базовом GameSettings

## Суть
`GameSettings` — абстрактный базовый класс, задаёт контракт и безопасные дефолты для всех режимов игры. Живой баланс (потолки, формулы опыта, стоимости, интервалы) — в конкретной активной реализации `EraSettings` (через DI). При правке баланса менять только `EraSettings` (и соседей `EraSettings5Lvl`, `EraSettingsV2..V4`, `TesSettings*`, если нужно). Базовый `GameSettings` — только для расширения контракта (новые методы/поля) и если меняется сам смысл API.

## Где в коде
- `src/app/GameSettings.ts` — абстрактный класс, сюда добавляются новые сигнатуры/константы, разделяемые всеми режимами (например, `perkPointLvlInterval`).
- `src/app/game-settings/EraSettings.ts` — активная реализация. Константы `maxAbilLvl`, `maxChaLvl`, `maxPersLevel`, `abPointsPerLvl`, формулы `getPersExpAndLevel`, `abChangeExp`, `abTotalCost`, `abCost`, `getMonsterLevel`, `checkPerkTskValue` — всё здесь.
- `src/app/game-settings/` — альтернативные реализации (`EraSettings5Lvl.ts`, `EraSettingsV2.ts`, `TesSettings*.ts` и т.п.); новые поля из `GameSettings` они игнорируют.

## Почему важно
Пользователь один раз явно откатил правку в `GameSettings` и перенёс её в `EraSettings` (задача 0016). Если сделать наоборот — сломается контракт для остальных реализаций и поменяется поведение «мёртвых» режимов. При добавлении нового поля балансной механики: если оно концептуально общее — заводим в `GameSettings` с безопасным дефолтом; конкретное значение и формулы — только в `EraSettings`.
