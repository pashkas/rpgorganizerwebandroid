# 0009. Попап изменений (`PerschangesService`) требует проброса поля в 3 места

## Суть
Всплывающий попап «что изменилось у перса» после действия с задачей строится через `PerschangesService`. Данные о состоянии задачи до/после собираются в `changesMap[tsk.id]`, оборачиваются в `PersChangesItem` и конвертируются в текст. Чтобы новое поле `Task` повлияло на текст/логику попапа, нужно пробросить его через **все три** места: (1) запись в `changesMap` при сборе состояния, (2) декларация поля в классе `PersChangesItem`, (3) использование в генерации строки (ветвление `txt`). Забыть хоть одно — поле будет `undefined` в попапе либо не повлияет на выбор ветки.

## Где в коде
- `src/app/perschanges.service.ts:548-549` — запись в `changesMap[tsk.id].isPerk`, `.perkHardnes` при сборе состояния.
- `src/app/perschanges.service.ts:167-176` — генерация текста: `if (el.isPerk) { txt = el.perkHardnes === 1 ? "перк активирован" : "перк прокачан"; }`.
- `src/app/perschanges.service.ts:667-668` — класс `PersChangesItem` с полями `isPerk`, `perkHardnes`.
- `src/app/perschanges.service.ts:457` — проброс `perkPoints: this.afterPers.OP` в `LevelUpMsgComponent` при level-up попапе (параллельный механизм для полей перса, не задачи).

## Почему важно
Любой новый флаг задачи, который должен менять текст/ветку попапа («перк активирован» vs «навык активирован»), требует проброса во все три места сразу. Разделение «что собрали» (`changesMap`) и «что есть в объекте для шаблона» (`PersChangesItem`) — не дублирование, а граница: первое — источник данных, второе — контракт для шаблона. Про level-up попап (`LevelUpMsgComponent`) отдельно — новые поля ресурсов перса (ОН, ОП, …) прокидываются туда напрямую через `perkPoints`-подобные ключи.
