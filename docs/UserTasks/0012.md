# 0012 — Simplify-ревью: очистка кода после серии доработок

## Запрос пользователя
Запуск `/simplify` на `pers.service.ts` и всех изменённых файлах — ревью на переиспользование, качество и эффективность.

## Цель
Найти и исправить проблемы качества кода, накопившиеся за серию доработок (чеклисты, дни недели подзадач, masonry→grid, свайпы, failCounter и т.д.).

## Что нашли и исправили

### 1. moveChecklistItem — ручной swap вместо CDK (`add-item-dialog.component.ts`)
`moveItemInArray` из `@angular/cdk/drag-drop` уже был импортирован, но не использовался. Ручная перестановка через tmp-переменную заменена на вызов CDK-утилиты.

### 2. addChecklistItem — plain object вместо класса (`add-item-dialog.component.ts`)
Создавался литерал `{ name, isDone: false }` вместо экземпляра `ChecklistItem`. Добавлен импорт `ChecklistItem` из `Task.ts`, используется `new ChecklistItem()`.

### 3. Неиспользуемые поля в customSwipe.directive.ts
- Поле `swiping` устанавливалось в `true`/`false`, но нигде не читалось — удалено.
- `ElementRef` инжектился через конструктор, но не использовался — удалён из импорта и конструктора.

### 4. `new Date(tsk.date)` внутри цикла (`pers.service.ts`)
В фильтре по дням недели `new Date(tsk.date)` создавался на каждой итерации цикла по `tsk.states`, хотя `tsk.date` инвариантен. Вынесен перед циклом.

### 5. Длинный объект data для диалога (`task-detail.component.ts`)
Объект из 9 полей (~220 символов) инлайном в вызове `dialog.open`. Вынесен в именованную переменную `dialogData`.

### 6. Пустая строка перед return в getChecklistText (`task-detail.component.ts`)
Лишняя пустая строка в однострочном методе — убрана.

## Что проверили, но не стали фиксить
- Дублирование сброса чеклистов (`task-detail` vs `setStatesNotDone`) — разная логика (второй также сбрасывает `isActive`).
- `isStateVisibleByDate` с `new Date` — не hot path, кэширование усложнит без заметного выигрыша.
- Паттерн `|| 1` в двух местах `pers.service.ts` — контексты разные, выносить в метод ради двух вхождений избыточно.
- Условие `(!qw.isNoActive || (qw.abilitiId && !qw.parrentId))` без комментария — бизнес-логика, комментарий был бы полезен, но это не баг.
