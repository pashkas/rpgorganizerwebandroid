# 0022 — Ограничение прокачки перков: только если нет доступных обычных навыков

## Запрос пользователя

> «Тут есть ограничение на прокачку навыков, что, например до 10-го уровня можно прокачать не более чем до второго уровня. И еще есть перки (норм и сложн). Я хочу добавить еще одно ограничение, что прокачивать перки можно только в том случае, если нет доступных навыков для прокачки (грубо говоря до 10-го уровня перса все на втором уровне). Но надо учесть, что если навык "сложн" и он открыт (уровень больше 0), его можно и даже нужно "докачать", там на эту тему есть проверка в persService»

## Цель

Перки должны качаться только когда база (обычные навыки) уже упёрлась в потолок по уровню перса. Идея: перк — это «вишенка» на прокачанной характеристике, а не параллельная ветка. Пока среди обычных навыков есть хоть один доступный к апгрейду (`mayUp=true`, с учётом всех предыдущих ограничений — ОН, открытости, потолка по уровню перса) — перки блокируются.

## Исключение

Начатый сложн-перк (`perkHardnes === 1 && value > 0`) должен остаться апаемым. Причина: сложн-перк прокачивается в два шага по 5 ОН, и если его не докачать, половина ОН «застрянет» в начатом перке. Норм-перк (`perkHardnes === 0.5`) прокачивается одним шагом сразу до `maxAbilLvl`, так что у него `value>0` уже означает «готов» — и он отсекается более ранним блоком по потолку.

## Подход

Новый блок (шаг 4) добавлен в метод `recountAbilMayUp` в [src/app/pers.service.ts](../../src/app/pers.service.ts) — сразу после блока ограничения по макс. уровню и перед пересортировкой:

```ts
const anyRegularMayUp = prs.characteristics.some((ch) =>
  ch.abilities.some((ab) => ab.tasks.some((t) => !t.isPerk && t.mayUp))
);

if (anyRegularMayUp) {
  // сбрасываем mayUp=false у всех перков, кроме начатого сложн
  // isHardOpened = (tsk.perkHardnes ?? 0.5) === 1 && tsk.value > 0
}
```

Порядок проверок критичен: `anyRegularMayUp` считается уже *после* того, как предыдущие блоки применили все ограничения к обычным навыкам. Если бы флаг считался раньше — мы бы учитывали навыки, которые на самом деле заблокированы по ОН или по потолку уровня, и ошибочно разрешали бы перки.

## Структура метода после изменений

`recountAbilMayUp` теперь явно разбит на 4 шага (с комментариями):

1. Базовые блокировки (ОН, `value < 1`) + расчёт `IsNextLvlSame` для начатых сложн-перков.
2. Если есть начатый сложн-перк — блокируем всё остальное (принуждаем докачать).
3. Потолок по уровню перса: обычные — `2 + floor(persLevel/10)*2`, перки — `maxAbilLvl`.
4. **НОВОЕ:** перки блокируются, если есть доступный обычный навык. Исключение — начатый сложн-перк.

Финальная пересортировка характеристик/навыков — апаемое всплывает наверх.

## Побочные эффекты / edge cases

- **Сортировка:** шаг 4 сбрасывает `mayUp=false` у перков, но `ch.anyMayUp` пересчитывается *раньше* (в шаге 1). После шага 4 он не обновляется — так же, как и после шага 3 с потолком по уровню. Это унаследованное поведение метода, не меняли. Для сортировки принципиально «есть ли где апать в характеристике», и перки в заблокированной ветке туда попадать не должны — но на практике, если у характеристики блокируются только перки, `anyMayUp` может быть завышен. Если всплывёт проблема — пересчитать после шага 4.
- **Норм-перк:** `value > 0` → он уже на `maxAbilLvl`, значит отсекается шагом 3. В шаге 4 проверка `isHardOpened` ложная → блокируется. Но это не важно, т.к. `mayUp` уже false.
- **Сложн-перк на `value = maxAbilLvl`:** `mayUp` сброшен на шаге 3, `isHardOpened=true`, но условие блокировки «`if (!isHardOpened)`» и не трогает его — остаётся false. OK.
- **Взаимодействие с шагом 2** (`anySame`): если есть начатый сложн-перк, шаг 2 уже заблокирует все обычные навыки → `anyRegularMayUp = false` → шаг 4 ничего не меняет. Это корректно: начатый перк важнее.
