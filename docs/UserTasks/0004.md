# 0004 — Белый экран из-за фичи «Дни недели» в подзадачах

## Запрос пользователя

> «У меня на телефоне что-то поломалось. Вчера работало и если установить дату вчерашнего дня, то тоже. А на сегодняшний день не работает. Может быть из-за дней недели в подзадачах? Потому что как раз на сегодня есть одна с этим включенным чекбоксом.»
>
> Далее: «На таком персе ломается», «у меня на телефоне вообще пустой экран (белый)», «Если дата установлена вчерашняя — всё норм», «Когда я удалил стейты из задачи "феншуй" (была в примере), всё заработало. Значит проблема в стейтах (подзадачах)».

## Цель задачи

Починить падение приложения (белый экран) после добавления новой фичи выбора дней недели (`tskWeekDays`) и чек-листов (`isChecklist`, `checklistItems`) для подзадач (`taskState`). Симптом наблюдался только на персонаже со стартовой задачей «фэншуй» у которой были подзадачи с новыми полями, и только когда системная дата совпадала с сегодняшним днём.

## В чём была проблема

Белый экран = необработанное исключение на старте. Корень — сочетание нескольких багов, внесённых вместе с фичей:

1. **Нет нормализации новых полей у подзадач.** В цикле загрузки перса ([pers.service.ts:1711-1728](src/app/pers.service.ts#L1711-L1728)) для `tsk.states[i]` не инициализировались `tskWeekDays`, `isChecklist`, `checklistItems`. Легаси-подзадачи (включая стартовые из SamplePers) приходили с `undefined`. Firestore к тому же не сериализует пустые массивы — даже свежесозданные объекты после round-trip возвращались с `undefined`.

2. **`getWeekKoef` не гвардил `undefined`.** [pers.service.ts:1238-1260](src/app/pers.service.ts#L1238-L1260): `let wd = weekDays.length` бросал `TypeError`, если `weekDays === undefined`. Вызов шёл из трёх мест ([:2191](src/app/pers.service.ts#L2191), [:2978](src/app/pers.service.ts#L2978), [:3357](src/app/pers.service.ts#L3357)). При `requrense === "дни недели"` и отсутствующем поле — крэш на старте до рендера.

3. **`CheckSetTaskDate` не гвардил `undefined`.** [pers.service.ts:116](src/app/pers.service.ts#L116): `tsk.tskWeekDays.length === 0` — такой же `TypeError` при `undefined`. Формально защищалось нормализацией выше по потоку, но при изменении порядка вызовов легко ломалось.

4. **Принудительный `push("пн")` в пустой `tskWeekDays`.** [pers.service.ts:1683-1685](src/app/pers.service.ts#L1683-L1685): на каждой загрузке в любую задачу с пустым массивом запихивался `"пн"`. Для задач с `requrense === "дни недели"` и пустым списком дней это означало, что `CheckSetTaskDate` прокручивал `tsk.date` вперёд до ближайшего понедельника — и задача уезжала из «сегодня». Объясняет эффект «в четверг работало, в пятницу — нет» для задач-расписаний (хотя основной крэш давали пункты 1-3).

5. **Пайп `requrence` мутировал вход через `.sort()`.** [shared/requrence.pipe.ts:17](src/app/shared/requrence.pipe.ts#L17) — сортировал `tskWeekDays` на месте, что недопустимо для Angular-пайпа.

## Подход и почему такой

**Точечные дефенсивные гварды + нормализация новых полей в цикле загрузки.** Без рефакторов, без замены мутаций, без миграций чистящих старые данные.

Мотивация:
- Корневой источник белого экрана — два непокрытых `.length` на `undefined`. Достаточно дефенсивных гвардов + нормализации в одном месте.
- Удаление `push("пн")` отдельным пунктом — устраняет логическую ошибку с уездом `tsk.date` в понедельник, но это вторичный баг (не крэш).
- `.sort()` в пайпе — мелкая чистка, на том же стеке.
- Пункт из плана про рефактор мутации `el.isActive` в `getPlusState` сознательно **не делали**, см. ниже в «побочных эффектах».

## Затронутые файлы и классы

**[src/app/pers.service.ts](src/app/pers.service.ts):**
- **`CheckSetTaskDate` (строка 116)** — добавлен гвард `!tsk.tskWeekDays || tsk.tskWeekDays.length === 0`.
- **`getWeekKoef` (строка 1251)** — `let wd = (weekDays && weekDays.length) || 0` вместо голого `weekDays.length`.
- **Нормализация `Task` (строки 1679-1685)** — удалён блок `if (tsk.tskWeekDays.length == 0) { tsk.tskWeekDays.push("пн"); }`. Оставлена только инициализация `= []`.
- **Нормализация подзадач (строки 1724-1733)** — в цикл `for (const st of tsk.states)` добавлены инициализации `st.tskWeekDays`, `st.isChecklist`, `st.checklistItems`.

**[src/app/shared/requrence.pipe.ts](src/app/shared/requrence.pipe.ts):**
- **`convertToRangeString` (строка 17)** — `[...(tskWeekDays || [])].sort(...)` вместо мутирующего `tskWeekDays.sort(...)`.

## Побочные эффекты и риски, которые учли

- **Мутация `el.isActive` в `getPlusState` ([pers.service.ts:3272-3289](src/app/pers.service.ts#L3272-L3289)) — оставлена как есть.** По плану был пункт 5 про её удаление, но грепом выяснилось: `isActive = true` выставляется **только** в этом методе, больше нигде. Если убрать, мгновенно ломаются:
  - `subtaskDoneOrFail` → `allIsDone` проверка на [:2491](src/app/pers.service.ts#L2491) (триггернёт `setTaskNextDate` сразу же);
  - счётчики активных сабтасков на [:2195](src/app/pers.service.ts#L2195), [:2457](src/app/pers.service.ts#L2457), [:3319](src/app/pers.service.ts#L3319).

  Мутация day-of-week внутри `getPlusState` семантически корректна (подзадачи не на сегодняшний день не должны считаться активными в `allIsDone`). Убирать её правильно только вместе с рефактором в отдельный метод `recalcStatesActivity`, что выходит за рамки точечной починки белого экрана.

- **Миграция старых данных, куда уже уехал `"пн"`, не делалась.** Пользователь подтвердил, что затронутые задачи поправит вручную.

- **Фильтр видимости подзадач по дням недели остался по `tsk.date`**, не по `new Date()` (решение пользователя при обсуждении плана). Сам эффект «не работает на сегодня» исчезает после фикса основного крэша — `tsk.date` у активной повторяющейся задачи остаётся на сегодня, поскольку баг с `push("пн")` больше не прокручивает её в понедельник.

## Корректировки от пользователя

В ходе задачи уточнения приходили порциями и сильно меняли картину:

1. Сначала описание: «не работает сегодня, но работает если выставить дату вчерашнего дня». Гипотеза была про логику фильтра `tsk.date`.
2. «На таком персе ломается» — намёк на data-specific проблему.
3. «У меня на телефоне вообще пустой экран (белый)» — картина сменилась с «плохо показывается» на «крэш на старте».
4. «Если дата вчерашняя — всё норм» — сигнал, что крэш зависит от `new Date()` / `getDay()`.
5. **«Когда я удалил стейты из задачи "феншуй" — всё заработало»** — решающая подсказка: проблема в обработке `tsk.states`, а не в самой задаче. Это и позволило сфокусироваться на негвардованных `.length` доступах к полям подзадач и на отсутствующей нормализации.

Также при обсуждении плана пользователь выбрал:
- Фильтровать подзадачи по `tsk.date`, а не по `new Date()`.
- Не делать миграцию старых данных.
- Рефакторить мутацию `el.isActive` в рамках этой задачи. По факту исполнения решил не трогать — см. «побочные эффекты» выше.
